## General
- Always follow the project's coding standards and style guide
- Always write clear and concise documentation for your code
- Always write unit tests for new features and bug fixes
- Always use meaningful variable and function names
- Always handle errors and exceptions gracefully

## Github - General
- Always create a new feature branch when starting work on a new feature or bug fix
- Always create a pull request (PR) for code reviews before merging into the main branch
- Always write clear and descriptive commit messages
- Always link pull requests to the relevant issue(s)
- Always sign commits

## Github Commit Messages
- Use imperative mood, e.g. "Add feature" instead of "Added feature"
- Use the format "type(scope): subject" for commit messages
- Use "BREAKING CHANGE" in the footer for breaking changes
- Use "Closes #issue_number" to link commits to issues
- Use "Refs #issue_number" to reference issues without closing them
- Use "See #issue_number" to reference issues without closing them
- Use "Related to #issue_number" for related issues

## Github Commit Types
- Use "feat" for new features,
- Use "fix" for bug fixes
- Use "chore" for maintenance tasks that don't affect functionality
- Use "style" for formatting changes that don't affect functionality
- Use "refactor" for code changes that neither fix a bug nor add a feature
- Use "test" for adding or updating tests
- Use "perf" for performance improvements
- Use "build" for changes that affect the build system or external dependencies
- Use "ci" for changes to CI configuration files and scripts
- Use "revert" for reverting previous commits
- Use "docs" for documentation-only changes
